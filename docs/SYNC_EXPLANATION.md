# Объяснение синхронизации пресетов в FilamentHub

## 🎯 Что мы хотим получить

**Цель:** Синхронизировать пресеты филаментов из FilamentHub в OrcaSlicer.

**Процесс:**
1. Пользователь нажимает кнопку "Synchronize" (или открывает вкладку FilamentHub)
2. Загружаем токен авторизации из настроек
3. Отправляем HTTP запрос к API: `GET /api/v1/auth/my-presets?updated_since=...`
4. Получаем список пресетов (JSON)
5. Для каждого пресета:
   - Если пресет новый (нет в маппинге) → скачиваем и импортируем
   - Если пресет уже есть (есть в маппинге), но API вернул его (значит обновлен) → переимпортируем
6. Обновляем `last_sync_time` (для следующей инкрементальной синхронизации)
7. Обновляем UI (кнопка, информация о пользователе)

## 🔧 Как это работает технически

### HTTP запросы (FilamentHubClient)

**Ключевой момент:** Используется `Http::perform_sync()`, а не `Http::perform()`.

```cpp
Http::get(url)
    .on_complete([on_complete](std::string body, unsigned status) {
        on_complete(body, status);  // Коллбэк вызывается В ТОМ ЖЕ ПОТОКЕ
    })
    .perform_sync();  // СИНХРОННЫЙ запрос - блокирует поток до завершения
```

**Важно:**
- `perform_sync()` вызывает `http_perform()` **НАПРЯМУЮ** в том же потоке
- Коллбэки (`on_complete`, `on_error`) вызываются **В ТОМ ЖЕ ПОТОКЕ**, где был вызван `perform_sync()`
- Если `perform_sync()` вызван из UI потока (главного потока wxWidgets), то коллбэки тоже в UI потоке

**Разница:**
- `perform_sync()` - синхронный, блокирует поток, коллбэки в том же потоке
- `perform()` - асинхронный, создает отдельный поток, коллбэки в другом потоке (тогда нужен `CallAfter`)

### Обновление UI из коллбэков

**В старой версии (работала):**
```cpp
client.get_my_presets(access_token, updated_since,
    [this, user_id](std::string json_body, unsigned http_status) {
        // Коллбэк выполняется в UI потоке (т.к. perform_sync() был вызван из UI потока)
        // Можно напрямую обновлять UI
        update_sync_button_state(false);  // ✅ Прямое обновление
        update_user_info();                // ✅ Прямое обновление
        wxMessageBox(...);                 // ✅ Прямое обновление
    }
);
```

**В новой версии (не работала):**
```cpp
client.get_my_presets(access_token, updated_since,
    [this, user_id](std::string json_body, unsigned http_status) {
        // Коллбэк выполняется в UI потоке, но использовали CallAfter
        CallAfter([this]() {
            update_sync_button_state(false);  // ❌ Излишне сложно
            update_user_info();
        });
    }
);
```

**Проблема:** `CallAfter` был добавлен из-за непонимания, что `perform_sync()` гарантирует выполнение коллбэков в том же потоке. Это усложнило код без необходимости.

## 📊 Счетчик `m_active_syncs` vs флаг `m_is_syncing`

### Старая версия (простая):
```cpp
bool m_is_syncing { false };  // Простой флаг

void on_sync_button_click() {
    if (m_is_syncing) return;  // Предотвращаем повторный клик
    update_sync_button_state(true);  // Устанавливаем флаг
    synchronize_presets(false);  // Запускаем синхронизацию
    // Флаг сбрасывается в коллбэке после завершения
}
```

**Логика:** Одна синхронизация (только presets), простой флаг, работает.

### Новая версия (сложная):
```cpp
bool m_is_syncing { false };
int m_active_syncs { 0 };  // Счетчик параллельных синхронизаций

void on_sync_button_click() {
    if (m_is_syncing) return;
    m_active_syncs = 0;  // Сбрасываем счетчик
    update_sync_button_state(true);
    synchronize_presets(false);      // Увеличивает счетчик после 200 OK
    synchronize_printer_profiles(false);  // Увеличивает счетчик после 200 OK
    synchronize_print_profiles(false);    // Увеличивает счетчик после 200 OK
    // Счетчик уменьшается в каждом коллбэке после завершения
    // Кнопка разблокируется, когда счетчик = 0
}
```

**Логика:** Три параллельные синхронизации (presets, printer profiles, print profiles), счетчик для отслеживания, сложнее.

**Проблема:** 
1. Счетчик увеличивается только после успешного ответа (200 OK)
2. Если запрос завершился с ошибкой (401, 403, 500), счетчик не был увеличен, но нужно разблокировать кнопку
3. Логика стала сложной и подверженной ошибкам
4. В старой версии была только одна синхронизация (presets), поэтому счетчик не был нужен

## 🔄 Уведомления: `wxMessageBox` vs `show_notification_in_webview`

### Старая версия:
```cpp
wxMessageBox(
    _L("Please login to FilamentHub first."),
    _L("FilamentHub Sync Error"),
    wxOK | wxICON_WARNING
);
```

**Плюсы:**
- Просто
- Модальное окно - пользователь обязательно увидит
- Работает везде

**Минусы:**
- Блокирует UI до закрытия
- Может быть навязчивым

### Новая версия:
```cpp
show_notification_in_webview(
    _L("Please login to FilamentHub first."),
    "warning"
);
```

**Плюсы:**
- Не блокирует UI
- Более современный подход
- Интегрировано в WebView

**Минусы:**
- Может быть незамеченным
- Требует WebView
- Сложнее в реализации

**Решение:** В старой версии использовали `wxMessageBox`, потому что это проще и надежнее. Для MVP это было достаточно.

## 📝 Обработка пустого списка пресетов

### Старая версия:
```cpp
if (presets.empty()) {
    BOOST_LOG_TRIVIAL(info) << "FilamentHub: No presets to sync";
    return;  // ❌ ПРОБЛЕМА: Не обновляем состояние кнопки!
}
```

**Проблема:** Если список пуст, кнопка остается в состоянии "Synchronizing..." навсегда.

**Исправление (нужно добавить):**
```cpp
if (presets.empty()) {
    BOOST_LOG_TRIVIAL(info) << "FilamentHub: No presets to sync";
    update_sync_button_state(false);  // ✅ Обновляем состояние кнопки
    return;
}
```

## 🎯 Вывод: Что нужно сделать

1. **Убрать `CallAfter`** - коллбэки уже в UI потоке, можно обновлять UI напрямую
2. **Убрать `m_active_syncs`** - оставить только `m_is_syncing`, упростить логику
3. **Использовать `wxMessageBox`** - проще и надежнее для MVP
4. **Исправить обработку пустого списка** - всегда обновлять состояние кнопки
5. **Упростить логирование** - убрать избыточные логи уровня `error`, использовать `info`
6. **Вернуться к простой логике** - как в старой версии, которая работала

## 🔍 Почему старая версия работала

1. **Простота:** Одна синхронизация (presets), простой флаг, прямое обновление UI
2. **Надежность:** Модальные окна для ошибок, всегда обновляется состояние кнопки
3. **Понятность:** Код легко читать и отлаживать
4. **Работает:** Не было сложной логики с счетчиками и `CallAfter`

## 🚀 Что мы хотим получить в итоге

**Простая, надежная синхронизация:**
1. Пользователь нажимает кнопку или открывает вкладку
2. Загружаем токен
3. Отправляем запрос к API (синхронно)
4. Получаем список пресетов
5. Импортируем новые/обновленные пресеты
6. Обновляем `last_sync_time`
7. Обновляем UI (кнопка, информация о пользователе)
8. Показываем ошибки через `wxMessageBox` (если есть)

**Без:**
- Сложных счетчиков
- `CallAfter` (не нужен для синхронных запросов)
- `show_notification_in_webview` (используем `wxMessageBox`)
- Избыточного логирования
- Синхронизации printer/print profiles (пока только presets)

**Результат:** Работающая синхронизация, как в старой версии, но с исправлением пустого списка.
