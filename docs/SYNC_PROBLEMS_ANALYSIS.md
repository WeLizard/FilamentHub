# Анализ проблем синхронизации OrcaSlicer ↔ FilamentHub

## 🔴 Критические проблемы

### 1. Двойное увеличение счетчика `m_active_syncs` в `synchronize_print_profiles`

**Проблема:**
```cpp
// Строка 2506: Увеличиваем счетчик ДО вызова API
m_active_syncs++;
BOOST_LOG_TRIVIAL(info) << "FilamentHub: Incremented m_active_syncs for print profiles. Active syncs: " << m_active_syncs;

// ... вызов API ...

// Строка 2552: Увеличиваем счетчик ЕЩЕ РАЗ после успешного ответа 200 OK
m_active_syncs++;
BOOST_LOG_TRIVIAL(info) << "FilamentHub: Incremented m_active_syncs for print profiles. Active syncs: " << m_active_syncs;
```

**Последствия:**
- Счетчик увеличивается дважды для print profiles
- При ошибке 401/403 счетчик остается увеличенным (уменьшается только при успешном ответе)
- UI может зависнуть, так как счетчик никогда не достигнет 0

**Исправление:**
Убрать увеличение счетчика на строке 2506, оставить только на строке 2552 (после успешного ответа 200 OK), как в `synchronize_printer_profiles`.

---

### 2. Несогласованная логика счетчика между `synchronize_printer_profiles` и `synchronize_print_profiles`

**Проблема:**
- `synchronize_printer_profiles`: счетчик увеличивается ТОЛЬКО после успешного ответа 200 OK (строка 2351)
- `synchronize_print_profiles`: счетчик увеличивается ДО вызова API (строка 2506) И после успешного ответа (строка 2552)

**Последствия:**
- Разная логика обработки ошибок
- При ошибке 401/403 в print profiles счетчик остается увеличенным
- При ошибке 401/403 в printer profiles счетчик не увеличивается (правильно)

**Исправление:**
Унифицировать логику: счетчик должен увеличиваться ТОЛЬКО после успешного ответа 200 OK в обоих методах.

---

### 3. Блокирующий характер `Http::perform_sync()`

**Проблема:**
`perform_sync()` - это синхронный вызов, который блокирует UI поток до получения ответа от сервера.

**Последствия:**
- UI зависает во время HTTP запросов
- Если запрос долгий (таймаут 30 секунд), UI может зависнуть на 30 секунд
- Коллбэки (`on_complete`, `on_error`) вызываются внутри `perform_sync()`, что означает, что они выполняются в UI потоке

**Текущая реализация:**
```cpp
Http::get(url)
    .header("Authorization", "Bearer " + access_token)
    .on_complete([on_complete](std::string body, unsigned status) {
        // Этот коллбэк вызывается внутри perform_sync()
        on_complete(body, status);
    })
    .perform_sync(); // Блокирует UI поток до получения ответа
```

**Рекомендация:**
- Использовать `Http::perform()` вместо `perform_sync()` для асинхронных запросов
- Или убедиться, что `perform_sync()` использует event loop и не блокирует UI

### 4. Неправильная обработка ошибок 401/403 в `synchronize_print_profiles`

**Проблема:**
```cpp
// Строка 2506: Увеличиваем счетчик ДО вызова API
m_active_syncs++;

// ... вызов API ...

// Строка 2530-2541: При ошибке 401/403 счетчик НЕ уменьшается
if (http_status == 401) {
    // Не уменьшаем счетчик - он не был увеличен для этой синхронизации
    return;
}

if (http_status == 403) {
    // Не уменьшаем счетчик - он не был увеличен для этой синхронизации
    return;
}
```

**Последствия:**
- Счетчик был увеличен на строке 2506, но при ошибке 401/403 он не уменьшается
- UI зависает, так как счетчик никогда не достигнет 0

**Исправление:**
Уменьшать счетчик при ошибке 401/403, если он был увеличен до вызова API.

---

### 5. Неправильная инициализация счетчика в `on_sync_button_click()`

**Проблема:**
```cpp
// Строка 1761: Инициализируем счетчик
m_active_syncs = 0;

// Строка 1791: Устанавливаем счетчик для filament presets
m_active_syncs = 1; // Filament presets (всегда)

// Строка 1802-1803: Вызываем синхронизацию printer/print profiles
synchronize_printer_profiles(false);
synchronize_print_profiles(false);
```

**Последствия:**
- `synchronize_printer_profiles` и `synchronize_print_profiles` могут увеличить счетчик позже (после ответа API)
- Но если они не увеличивают счетчик (например, при ошибке 401/403), счетчик остается равным 1
- UI может зависнуть, если filament presets синхронизация завершится, но счетчик не будет правильно обновлен

**Рекомендация:**
Счетчик должен увеличиваться ТОЛЬКО внутри методов синхронизации, а не в `on_sync_button_click()`.

## 🟡 Проблемы средней важности

### 6. Отсутствие проверки разрешений перед синхронизацией printer/print profiles

**Проблема:**
`on_sync_button_click()` вызывает `synchronize_printer_profiles()` и `synchronize_print_profiles()` без проверки разрешений пользователя.

**Последствия:**
- Лишние HTTP запросы, если разрешения отключены
- Пользователь видит ошибки 403, хотя синхронизацию можно было пропустить

**Рекомендация:**
Восстановить вызов `check_user_permissions()` перед синхронизацией printer/print profiles, как было раньше.

---

### 7. Неправильная обработка пустых списков

**Проблема:**
В `synchronize_print_profiles` при пустом списке счетчик уменьшается дважды:
- Строка 2506: счетчик увеличивается ДО вызова API
- Строка 2566: счетчик уменьшается при пустом списке
- Но счетчик также был увеличен на строке 2552 после успешного ответа 200 OK

**Последствия:**
- Счетчик может стать отрицательным или неправильным

**Исправление:**
Унифицировать логику: счетчик должен увеличиваться ТОЛЬКО после успешного ответа 200 OK, и уменьшаться только один раз.

---

## 🟢 Мелкие проблемы

### 8. Избыточное логирование

**Проблема:**
Много `BOOST_LOG_TRIVIAL(error)` для информационных сообщений, что засоряет логи.

**Рекомендация:**
Использовать правильные уровни логирования:
- `error` - только для ошибок
- `warning` - для предупреждений
- `info` - для информационных сообщений
- `debug` - для отладочных сообщений
