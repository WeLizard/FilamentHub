# Объяснение системы токенов в FilamentHub

## 🎯 Да, токен формируется отдельно для каждого пользователя

### Как работает система токенов

#### 1. Генерация токена при логине

**Код:** `backend/app/api/v1/endpoints/auth.py`

```python
@router.post("/login", response_model=Token)
async def login(...):
    # Проверяем пользователя
    user = await db.execute(select(User).where(User.email == data.email))
    
    # Создаем токен с данными пользователя
    token_data = {
        "sub": user.email,      # Email пользователя (subject)
        "user_id": user.id,     # ID пользователя
        "role": user.role.value # Роль пользователя
    }
    access_token = create_access_token(data=token_data)
    refresh_token = create_refresh_token(data=token_data)
    
    return Token(access_token=access_token, refresh_token=refresh_token)
```

**Что происходит:**
- Токен содержит уникальные данные пользователя: `email`, `user_id`, `role`
- Каждый пользователь получает свой токен с его данными
- Токен генерируется при каждом логине (даже для того же пользователя)

#### 2. Структура JWT токена

**Что внутри токена:**
```json
{
    "sub": "user@example.com",    // Email пользователя (subject)
    "user_id": 1,                 // ID пользователя
    "role": "user",               // Роль пользователя
    "exp": 1705320000,            // Время истечения (timestamp)
    "type": "access"              // Тип токена
}
```

**Важно:**
- Токен содержит всю информацию о пользователе внутри себя
- Токен не хранится в базе данных (stateless JWT)
- Токен подписывается секретным ключом (`SECRET_KEY`) для защиты от подделки

#### 3. Проверка токена

**Код:** `backend/app/core/dependencies.py`

```python
async def get_current_user(...):
    # Декодируем токен
    payload = decode_access_token(token)
    
    # Получаем email из токена
    email = payload.get("sub")
    
    # Находим пользователя в базе данных
    user = await db.execute(select(User).where(User.email == email))
    
    return user
```

**Что происходит:**
1. Токен декодируется и проверяется подпись
2. Из токена извлекается `email` (subject)
3. Пользователь находится в базе данных по `email`
4. Проверяется, что пользователь активен

#### 4. Хранение токена

**В OrcaSlicer:**
- Токен хранится в `AppConfig` (локальный файл конфигурации)
- Секция: `filamenthub`
- Ключ: `access_token`
- Значение: JWT токен (строка)

**В FilamentHub (Frontend):**
- Токен хранится в `localStorage` браузера
- Ключ: `access_token`
- Значение: JWT токен (строка)

**В базе данных:**
- Токен НЕ хранится в базе данных (stateless JWT)
- Только данные пользователя хранятся в базе данных

## 🔑 Ключевые особенности

### 1. Токен уникален для каждого пользователя

**Почему:**
- Токен содержит `user_id` и `email` пользователя
- Каждый пользователь получает токен со своими данными
- Токен не может быть использован другим пользователем

**Пример:**
```
Пользователь 1 (ID=1, email=user1@example.com):
  Токен: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
  Содержит: {"sub": "user1@example.com", "user_id": 1, "role": "user"}

Пользователь 2 (ID=2, email=user2@example.com):
  Токен: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
  Содержит: {"sub": "user2@example.com", "user_id": 2, "role": "user"}
```

### 2. Токен создается при каждом логине

**Поведение:**
- При каждом логине создается новый токен
- Старые токены остаются валидными до истечения срока действия
- Пользователь может иметь несколько активных токенов (если логинился с разных устройств)

**Пример:**
```
Пользователь логинится в OrcaSlicer:
  Токен 1: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... (срок действия: 7 дней)

Пользователь логинится в браузере:
  Токен 2: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... (срок действия: 7 дней)

Оба токена валидны одновременно (до истечения срока действия)
```

### 3. Токен не хранится в базе данных

**Почему stateless JWT:**
- Токен содержит всю информацию о пользователе внутри себя
- Сервер не хранит токены в базе данных
- Сервер проверяет токен по подписи (используя `SECRET_KEY`)

**Преимущества:**
- Не нужно хранить токены в базе данных
- Масштабируемость (не нужно синхронизировать токены между серверами)
- Простота (не нужно управлять сессиями)

**Недостатки:**
- Нельзя отозвать токен до истечения срока действия
- Токен остается валидным до истечения срока действия (даже если пользователь вышел)

### 4. Срок действия токена

**Access Token:**
- По умолчанию: 7 дней (10080 минут, настраивается в `ACCESS_TOKEN_EXPIRE_MINUTES`)
- Используется для доступа к API
- Длинный срок действия для удобства (особенно для OrcaSlicer интеграции)

**Refresh Token:**
- По умолчанию: 30 дней (настраивается в `REFRESH_TOKEN_EXPIRE_DAYS`)
- Используется для обновления access token
- Длинный срок действия для удобства

**Код:** `backend/app/core/config.py`

```python
class Settings(BaseSettings):
    ACCESS_TOKEN_EXPIRE_MINUTES: int = 60 * 24 * 7  # 7 дней (10080 минут)
    REFRESH_TOKEN_EXPIRE_DAYS: int = 30             # 30 дней
    SECRET_KEY: str                                 # Секретный ключ для подписи токенов
    ALGORITHM: str = "HS256"                        # Алгоритм подписи
```

**Примечание:** 
- Access token имеет срок действия 7 дней для удобства пользователей OrcaSlicer
- Пользователь не будет вынужден логиниться каждый раз при открытии слайсера
- Refresh token имеет срок действия 30 дней для длительной работы без повторного входа

## 🔒 Безопасность

### 1. Подпись токена

**Как работает:**
- Токен подписывается секретным ключом (`SECRET_KEY`)
- Сервер проверяет подпись при декодировании токена
- Если подпись неверна, токен отклоняется

**Код:** `backend/app/core/security.py`

```python
def create_access_token(data: dict[str, Any]) -> str:
    to_encode = data.copy()
    expire = datetime.utcnow() + timedelta(minutes=settings.ACCESS_TOKEN_EXPIRE_MINUTES)
    to_encode.update({"exp": expire, "type": "access"})
    encoded_jwt = jwt.encode(to_encode, settings.SECRET_KEY, algorithm=ALGORITHM)
    return encoded_jwt

def decode_access_token(token: str) -> dict[str, Any] | None:
    try:
        payload = jwt.decode(token, settings.SECRET_KEY, algorithms=[ALGORITHM])
        if payload.get("type") != "access":
            return None
        return payload
    except JWTError:
        return None
```

### 2. Проверка пользователя

**Что проверяется:**
- Токен валиден (подпись правильная, не истек)
- Пользователь существует в базе данных
- Пользователь активен (`user.active == True`)

**Код:** `backend/app/core/dependencies.py`

```python
async def get_current_user(...):
    payload = decode_access_token(token)
    if payload is None:
        raise HTTPException(status_code=401, detail="Could not validate credentials")
    
    email = payload.get("sub")
    user = await db.execute(select(User).where(User.email == email))
    
    if user is None:
        raise HTTPException(status_code=401, detail="User not found")
    
    if not user.active:
        raise HTTPException(status_code=403, detail="User is inactive")
    
    return user
```

### 3. Защита от подделки

**Как защищено:**
- Токен подписывается секретным ключом (`SECRET_KEY`)
- Секретный ключ известен только серверу
- Невозможно подделать токен без знания секретного ключа

**Важно:**
- Секретный ключ должен быть надежным (случайная строка)
- Секретный ключ не должен быть в открытом доступе
- Секретный ключ хранится в переменных окружения (`.env`)

## ✅ Итоговый ответ

**Да, токен формируется отдельно для каждого пользователя:**

1. **Уникальность:** Каждый токен содержит уникальные данные пользователя (`user_id`, `email`, `role`)
2. **Изоляция:** Токен одного пользователя не может быть использован другим пользователем
3. **Безопасность:** Токен подписывается секретным ключом, защищая от подделки
4. **Stateless:** Токен не хранится в базе данных, содержит всю информацию внутри себя

**Особенности:**
- Токен создается при каждом логине (новый токен для каждого логина)
- Старые токены остаются валидными до истечения срока действия
- Пользователь может иметь несколько активных токенов (с разных устройств)
- Токен не хранится в базе данных (stateless JWT)

**Хранение:**
- В OrcaSlicer: `AppConfig` (локальный файл конфигурации)
- В FilamentHub (Frontend): `localStorage` браузера
- В базе данных: НЕ хранится (stateless JWT)
