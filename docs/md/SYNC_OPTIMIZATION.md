# Оптимизация синхронизации пресетов с OrcaSlicer

## Проблема
- Постоянные запросы синхронизации создают нагрузку на API и бэкенд
- Создаются дубликаты при повторной синхронизации
- Нет проверки, что синхронизацию уже делали

## Решение

### 1. Поле `last_sync_at` в User
- Отслеживает время последней успешной синхронизации
- Обновляется после успешного импорта пресетов

### 2. Инкрементальная синхронизация
- OrcaSlicer отправляет `updated_since` параметр (время последней синхронизации)
- Backend возвращает только пресеты, изменённые после этого времени
- Это снижает объём данных и ускоряет синхронизацию

### 3. Предотвращение дубликатов
- Используем `fhub_id` (FilamentHub ID) для обновления существующих пресетов
- Если `fhub_id` не указан, ищем по `name + filament_id + user_id`
- Сравнение хеша содержимого для определения изменений

### 4. Флаг `m_is_syncing` в OrcaSlicer
- Уже реализован в C++ коде
- Предотвращает параллельные синхронизации
- Не позволяет запускать синхронизацию, если она уже идёт

## Реализация

### Backend:
1. Добавить поле `last_sync_at: datetime | None` в модель `User`
2. Обновлять `last_sync_at` после успешной синхронизации
3. Добавить параметр `updated_since` в эндпоинты для фильтрации

### OrcaSlicer (C++):
- Уже реализовано: `m_is_syncing` флаг
- Уже реализовано: `last_sync_time` в AppConfig
- Уже реализовано: отправка `updated_since` параметра

## Итог
Основная оптимизация уже реализована на стороне OrcaSlicer. Нужно только:
1. Добавить `last_sync_at` в User для отслеживания на бэкенде
2. Обновлять это поле после успешной синхронизации
